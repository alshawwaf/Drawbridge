Pick the Service type — Check Point isn’t only TCP/UDP. Port-based (TCP/UDP/SCTP) takes a port; ICMP, RPC, DCE-RPC, GTP, Compound/Citrix TCP, “Other” (an IP protocol) and a services group are matched by name (correlated to the real predefined object); Application/Site matches an app. The engine reasons about each kind correctly — distinct protocols never falsely overlap a port.

Source and destination aren’t only IPs. Switch either type to a Domain (e.g. alshawwaf.ca, or .alshawwaf.ca for sub-domains), Access role, Dynamic object, Updatable object or Security zone to ask, say, “does host 10.1.1.222 have access to the domain alshawwaf.ca?”. Each object kind is its own match space, so the engine reasons in identity terms — a domain matches a rule’s domain object (exact or a parent), and is never confused with an IP cell. Domains and dynamic objects are created if missing; access-roles, security-zones and updatable-objects are reused (defined in Identity Awareness / topology / Check Point’s repository).

How it decides — behind the scenes

Every request runs the same first-match decision tree the market leaders use. It reuses a rule that already permits the traffic; widens a rule only when it matches the request exactly in the other two columns (so it never grants access nobody asked for); creates a least-privilege rule when nothing fits; and refuses to touch policy it can’t safely reason about. It compares each source/destination by kind — an IP/CIDR in address space, or a typed object (domain, access-role, dynamic, updatable, security-zone) in its own identity space — so a domain is matched against a rule’s domain object, never confused with an IP.
